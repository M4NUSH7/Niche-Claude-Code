# 06 - Memory Vault and Context Packs

Two complementary systems: **memory/** is long-term truth (written rarely, curated);
**context/** is session-start loading material (read at every session, per terminal).

## memory/ - the Obsidian vault (single source of truth)

```
memory/
+-- INDEX.md                     # hub: wikilinks to everything + dated "Active Context" block
+-- architecture/decisions.md   # locked decisions with rationale (e.g. "Tauri 2, not Electron")
+-- agents/index.md              # agent table + effort ceiling + dynamic-effort guidelines
+-- skills/index.md              # skill registry mirror
+-- build/{phases,t1..tN}.md     # build-state notes per terminal
\-- decisions/*.md               # domain decision notes (payment-strategy, tech-decisions, ...)
```

Rules:
- **No memory files outside `memory/`.** Ever. Docs may summarize; memory is authoritative.
  (One declared exception: `.agents/logs/tN.db` per-terminal shards + the derived
  `.agents/master.db` hold *agent working logs*, not decisions - a different artifact with a
  hard boundary. See `10_token_efficiency.md` Sec. 4 and `05_hooks_and_logging.md`'s
  "Per-terminal memory shards + derived master.db (D1)".)
- `INDEX.md` carries an **Active Context** block with dated status lines
  (e.g. "T4 Phase 0 COMPLETE (2026-07-14) ... T1 Phase 0: NOT yet done") - the cross-terminal
  situational awareness snapshot.
- Navigation is wikilinks/backlinks; `.obsidian/` (repo root) configures the vault to open at
  `INDEX.md`. Obsidian is the *navigation layer*, never a second source.
- Every architecture change appends to `memory/architecture/decisions.md` in the same commit.
- **Decisions are append-only, but not immortal.** A decision that gets reversed is not deleted  - 
  it is *superseded* by a pivot note that links back to it (`11_pivots.md`). Live and dead
  decisions must never look identical.

## Writing conventions (vendored, not installed)

The precise syntax that makes the vault navigable - audited out of `kepano/obsidian-skills` and
vendored here rather than installed as a skill (it would cost ~1,400 tokens on the most frequent
write path in the harness; see `10_token_efficiency.md` Sec. 5):

```markdown
[[Note]]              link          [[Note#Heading]]     section link
[[Note#^block-id]]    BLOCK link - cite one exact line, not a whole file
[[Note|alias]]        display text
```

Frontmatter is **typed** - use the types, not prose:

```yaml
---
date: 2026-05-02                       # date  -> sorts/filters natively
status: pivoted                        # text
supersedes: "[[2026-03-11-lstm]]"      # LINK  -> auto-generates a backlink (quote it: YAML)
salvaged: [ipc-layer, build-cfg]       # list
approved_by: <human>                   # text  -> enforces principle #11
aliases: [lstm-decision]               # reach a note by its informal name, no duplicate stub
tags: [decision/pivot]                 # nested tags: hierarchy without moving the file
---
```

The `supersedes:` link is load-bearing: lineage is stored **once**, on the pivot record, and the
abandoned note learns it was superseded **without being edited**. That is principle #4 working for
you - one source, navigation derived.

**Never hand-author a status/table view of the vault** (e.g. an Obsidian `.base` file). It looks
generated but drifts silently - audit #7's forbidden second bookkeeping system. Status blocks are
generated by `harness_status.py --write`.

## Pivots

`memory/decisions/` holds pivot notes: human-approved records of work that was abandoned or
refactored away. Template: `template/memory/pivot-template.md`. Doctrine, runbook, and the
`- [~]` / gate-revocation mechanics: **`11_pivots.md`**.

## Handoff folder (D2)

```
memory/handoff/
+-- {tN}/{YYYY-MM-DD}-{topic}.md      # per-terminal handoff notes
\-- session/{YYYY-MM-DD}-{topic}.md   # cross-terminal / whole-session handoff notes
```

A handoff note is the short, dated "what you need to know before you pick this up" message one
session leaves for the next - narrower than a pivot (no approval gate, no supersession), broader
than a checkbox (carries context a one-line phase item can't hold: what was tried, what's still
open, what to watch for). It is written by the outgoing session, human or agent, and named
`{YYYY-MM-DD}-{topic}.md` so newest sorts last.

**Session-start ritual, updated:** load `context/agents/tN-context.md` -> **read the newest file
in `memory/handoff/tN/` and `memory/handoff/session/` (if any)** -> open `.bld/tN/phases.md` ->
first unchecked item is the task. The re-read step goes *between* the context pack and the phase
file: the context pack is static (who you are), the handoff is dynamic (what just happened).
Skipping it means every session re-derives state the previous session already worked out.

`scaffold.sh` creates `memory/handoff/session/` and one `memory/handoff/{tN}/` per terminal.

## The fourth checkbox state - `- [!]` blocked/awaiting-human

Three states cover forward progress, done, and abandoned. They have no home for **"stuck, not
abandoned - needs a human decision."** A fourth mark closes that gap:

| Mark | Meaning | Counted as |
|---|---|---|
| `- [ ]` | open | the queue |
| `- [x]` | done, still standing | progress |
| `- [~]` | superseded (pivoted away from) | neither - links a pivot note |
| `- [!]` | **blocked / awaiting-human** | neither - links a decision note |

```markdown
- [!] Payment provider integration -> see memory/decisions/2026-07-20-provider-tbd.md
```

Use `- [!]` for a task that cannot proceed without a human call (an ambiguous requirement, a
missing credential, a business decision) - distinct from `- [~]`, which means the work itself was
judged wrong. A `- [!]` line should link a `memory/decisions/` note the same way a pivot does, so
"why is this stuck" has an answer instead of a silently stalled checkbox. This is documentation +
convention in this pass, not a `harness_status.py` reconciliation rule - see F1's goal-packet
doctrine (`12_graph_of_loops.md`) for how blocked/ambiguity terminals route into this same
mechanism from the agent side.

## context/ - per-terminal session-start packs

```
context/
+-- README.md                # rule: load ONLY your terminal's context file
+-- agents/tN-context.md     # per terminal: you own / current phase / your agents /
|                            #   key files / gate you control / unblock conditions /
|                            #   security-sensitive paths
+-- architecture/
|   +-- system-overview.md   # 1-page stack + diagram + module->schema map + hard rules
|   +-- module-map.md
|   \-- security-rules.md    # auth gates, append-only rules, webhook/HMAC rules, admin rules
+-- skills/index.md          # skill loading table + protocol
\-- .initial/                # full launch/strategy plan - REFERENCE-ONLY, never edited
```

Session-start ritual (from CLAUDE.md): read your `tN-context.md` -> open `.bld/tN/phases.md` ->
first unchecked item is the task. Nothing else loads by default.

## Division of labor

| Question | Answered by |
|---|---|
| "What did we decide and why?" | `memory/architecture/decisions.md` |
| "What is the state of the build right now?" | `memory/INDEX.md` Active Context + `.bld/*/phases.md` checkboxes |
| "What do I need to know to start working in T3?" | `context/agents/t3-context.md` |
| "What's the product/launch plan?" | `context/.initial/` (read-only) |
| "What happened, mechanically?" | `.claude/logs/{agent}/{agent}.db` |

## Caution learned from Coursly (see 09_AUDIT.md)

Keep **credentials out of context packs** (Coursly's T3 context embeds payment sandbox creds  - 
convenient, bad habit), and treat the Active Context block as generated-from-checkboxes rather
than hand-maintained, or it will drift from reality (Coursly's did).
